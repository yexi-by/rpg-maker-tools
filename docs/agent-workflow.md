# Agent 翻译流程概览

本文档给用户快速理解流程。Agent 的完整执行规程在 `skills/rpg-maker-translation/SKILL.md`。

## 总流程

```text
准备输入
→ Agent 读取 SKILL.md
→ 检查项目和游戏
→ 准备 Agent 工作区
→ 校验并导入自定义占位符规则
→ 当前 Agent 检查术语表、插件规则、事件指令规则
→ 执行翻译
→ 生成质量报告
→ 质量通过后写回
→ 用户实机抽查
```

## 用户负责什么

- 提供项目目录。
- 提供游戏根目录。
- 提供外部临时目录。
- 在终端环境变量中配置模型服务地址和 API Key。
- 决定是否允许最终写回。
- 在 Agent 遇到阻断问题时做选择。
- 写回后进行实机抽查。

## Agent 负责什么

- 按 `skills/rpg-maker-translation/SKILL.md` 执行。
- 检查项目环境和游戏结构。
- 识别这是首次翻译还是已有数据库的二次/增量翻译。
- 判断自定义控制符规则是否完整。
- 使用 `prepare-agent-workspace` 生成分析工作区，优先读取工作区内的 `placeholder-rules.json` 草稿。
- 用 `validate-placeholder-rules` 验证规则能解析、能生成语义化占位符，并确认样本文本可以完整还原。
- 当数据库中术语表、插件规则或事件指令规则为空时，导出对应文件，自行分析。
- 对游戏中实际存在的内容生成导入 JSON 并写入数据库。
- 对游戏中不存在的内容记录“已检查，确认无对应内容”，不要编造规则。
- 导出结果为空、插件没有玩家可见文本、事件指令参数没有玩家可见文本，都是合法结论。
- 如果 Agent 不确定结构含义，应暂停反馈，而不是把不确定当作无内容。
- 如果平台支持子代理，可以并行分析术语表、插件规则和事件指令规则；当前 Agent 仍负责复核、导入、验收和清理。
- 不整文件读取大型导出 JSON。导出和导入必须走本项目 CLI；中间分析可以使用 Agent 自带工具、临时脚本或子代理。
- 验收术语表、插件规则和事件指令规则已成功写入数据库，或已确认当前游戏没有对应内容。
- 删除本轮导出和分析产生的临时文件。
- 执行翻译和质量报告。
- 遇到阻断问题时暂停并反馈。
- 在质量报告通过且用户允许后写回。

## 二次翻译

已有数据库时，Agent 不应把任务当成全新游戏无脑重跑。应先执行 `doctor`、`quality-report`、`translation-status` 和占位符扫描，确认已有规则、缓存、运行故障、译文质量错误和质量风险。已有译文缓存会复用；CLI 只会处理当前提取范围内尚未成功入库的条目。二次写回会直接替换当前激活文件，原始文件留档不重复改写。

## CLI 负责什么

- 读取标准 RPG Maker MZ 数据。
- 管理每个游戏的数据库。
- 提取正文、插件文本和事件指令文本。
- 调用模型并缓存译文。
- 校验占位符、日文残留和长文本宽度。
- 写回游戏文件并建立留档。

## 阻断时应该发生什么

如果项目目录不存在、游戏数据不可解析、模型不可用、自定义控制符未覆盖、强制规则数据未检查、规则导入失败、翻译多轮失败、质量报告有阻断错误或用户未授权写回，Agent 应停止当前阶段并向用户报告。

报告应包含：当前阶段、已执行内容、阻断原因、关键证据、建议下一步和需要用户确认的信息。
