# 项目局部规范

本规范约束 `att-mz` 仓库内的提示词、临时文件、插件规则、文本规则、日志和交付行为。未覆盖事项继续遵循用户级通用规范。

## 0. 对用户说人话

- 面向用户的回复、报告、Skill 主流程、README、默认日志摘要和故障说明，必须直接说明“发生了什么、影响什么、下一步做什么”。
- 禁止用内部字段名或英文缩写代替解释。命令名、JSON 字段名和错误码可以保留，但第一次出现必须紧跟中文解释。
- 禁止把 `pending` 直接说给用户；统一说“还没成功保存译文的文本”。
- 禁止把 `quality_error` 直接说给用户；统一说“模型翻了，但项目检查没通过的译文”。
- 禁止把 `overwide_line` 直接说给用户；统一说“某一行太长，游戏窗口放不下”。
- 禁止把 `placeholder` 直接说给用户；统一说“必须原样保留的游戏控制符”。
- 禁止把 `write-back` 直接说给用户；统一说“把译文写进游戏文件”。
- 禁止把 `location_path` 直接说给用户；统一说“文本在游戏里的内部位置”。只有在 JSON 文件格式说明中才允许使用字段名。
- 禁止把 `translation_lines` 直接说给用户；统一说“中文译文行”。只有在 JSON 文件格式说明中才允许使用字段名。
- 禁止把 `run_id` 直接说给用户；统一说“本次翻译编号”。只有排障报告需要精确定位时才附带字段名。
- 禁止把“入库”“缓存”“门禁”“阻断”“产物”“收尾”“跑批”“去重后”“导出骨架”当作默认用户文案。必须改成“保存到项目数据库”“已保存的译文记录”“检查没通过，不能继续写进游戏文件”“生成的文件”“处理剩下的文本”“分组发送给模型”“相同原文只翻一次”“生成可填写的修复表”。
- 如果需要讨论内部实现，先给一句人话结论，再补内部原因。禁止先甩数据库表、字段名、类名、函数名。

## 1. 提示词规范

- 给 LLM 的提示词只能包含模型完成当前任务所需的信息。
- 禁止把仅供程序定位、调试、读写文件的内部实现细节塞进 prompt。
- 禁止在 prompt 中暴露模型无法访问也无法使用的来源说明，例如术语表工作区文件名、数据库表名、内部字段名、文件路径、`location_path`、`translated_text`。
- 所有给 LLM 的 prompt 都只能提供当前任务的可见输入、输出格式、质量要求和必要的原样保留约束；任何项目内部机制都不得进入 prompt。
- 正文翻译 prompt 只能要求模型按输出协议翻译，并原样保留 `[RMMZ_...]`、`[CUSTOM_...]` 等占位符；不得解释占位符含义、保护方式、恢复方式或写回机制。
- 术语表必须使用翻译领域常用称呼 `[[术语表]]`。
- 术语表条目只允许呈现为 `原文 => 标准译名`。
- 术语表条目禁止附加位置、文件名、JSON 字段名、数据库主键等内部定位信息。
- 内部路径和定位信息只能用于程序筛选术语、写回 `101.parameters[4]`、写回 `MapXXX.displayName`、日志和排障。
- 只有当模型必须原样回传某个标识以便程序绑定输出时，才允许在 prompt 中放置内部 ID；内部 ID 必须放在明确的结构化输出协议中。

## 2. 外部临时文件规范

- `101` 名字框、`MapXXX.displayName`、数据库名称和系统类型的译名由主代理审查后的术语表 JSON 提供。
- 面向外部代理的 JSON 必须使用最小结构：能用 `原文: 译名` 表达的内容不得拆成 ID、位置数组、类型字段和说明字段。
- 外部规则文件不得包含版本号、游戏标题、规则 ID、原因说明、内部路径、上下文文件名等主流程不需要读取的字段。
- 术语表 JSON 是项目外临时文件，必须通过 `import-terminology --game ... --input ...` 显式导入当前游戏数据库。
- 插件配置 JSON 是项目外临时文件，必须通过 `export-plugins-json --game ... --output ...` 从当前游戏导出。
- 插件配置 JSON 的顶层必须是 RPG Maker `$plugins` 数组本体，不得添加候选字段、说明字段或判断结果。
- 插件规则 JSON 是项目外临时文件，必须通过 `import-plugin-rules --game ... --input ...` 显式导入当前游戏数据库。
- 事件指令参数 JSON 是项目外临时文件，必须通过 `export-event-commands-json --game ... --output ...` 从当前游戏导出。
- 事件指令规则 JSON 是项目外临时文件，必须通过 `import-event-command-rules --game ... --input ...` 显式导入当前游戏数据库。
- 主流程的数据源必须是当前游戏数据库，`translate`、`write-back`、`run-all` 和提示词组装流程不得直接读取项目外临时 JSON。
- 多游戏数据必须按 `data/db/<游戏标题>.db` 分库存储。
- 不同游戏的术语表、插件规则和已保存译文记录不得混放在项目固定目录中作为运行时数据源。

## 3. 插件规则规范

- `plugins.js` 可翻译字段由外部代理产出规则 JSON。
- 插件规则导入时必须校验插件名称、插件哈希和 JSONPath 字符串叶子命中结果。
- 插件规则导入后只能把校验后的规则条目写入数据库。
- 导入流程不得把外部临时文件路径保存为长期业务数据源。

## 4. 事件指令规则规范

- `data/*.json` 中的事件指令参数文本统一由外部代理产出规则 JSON。
- 事件指令参数 JSON 未传 `--code` 时必须读取 `setting.toml` 的 `event_command_text.default_command_codes` 数组。
- 事件指令参数 JSON 传入 `--code` 时必须只使用 CLI 编码数组，并覆盖配置文件默认编码数组。
- 事件指令规则导入时必须校验指令编码、参数过滤条件和 JSONPath 字符串叶子命中结果。
- 事件指令规则必须显式声明 `command_code`，不得在业务代码中按固定编码猜测可翻译字段。
- 事件指令规则导入后只能把校验后的规则条目写入数据库。
- 主翻译流程只能按数据库事件指令规则提取复杂参数文本。

## 5. 文本规则配置规范

- RMMZ 标准文本控制符属于引擎协议，必须由代码内置并完整测试覆盖。
- 程序只能内置引擎公开协议和结构性校验，不能把启发式猜测当作游戏私有语义的最终判断。插件自定义控制符边界、脚本标记含义、可见文本字段、专名是否保留、插件路径是否可翻译等判断，必须来自用户明确输入、当前游戏文件的人工或外部代理分析、已导入规则、术语表或源文残留例外规则。
- 程序可以扫描并报告疑似候选，但候选只能用于提示和校验，不能自动升级为已确认规则。遇到语义边界不明确的候选时，必须输出可理解的告警或错误，让主代理补充外部判断；禁止靠程序自身猜测继续翻译、导入规则或消耗模型额度。
- 每次对真实游戏执行翻译测试、回写测试或质量验证前，必须先扫描当前游戏文本，找出完整的游戏自定义控制符、脚本标记和特殊占位符，并把对应自定义正则规则导入当前游戏数据库，或通过 CLI `--placeholder-rules` 传入。
- 自定义占位符规则确认完成前，禁止启动会消耗模型额度的正文翻译任务。
- 角色名牌、语音触发标记、自动替换触发标记和其他会影响游戏运行协议的可见前缀，必须优先按“必须原样保留的游戏控制符”处理。例如形如 `◆<角色名>ｔ` 的发声/名牌触发文本，不能默认当普通角色名翻译；必须先通过占位符规则校验和覆盖扫描，再启动会消耗模型额度的正文翻译任务。
- 如果在已有大量译文成功保存后，才发现角色/语音触发标记被当作普通文本处理，必须立刻暂停并向用户说明已保存译文数量、疑似受影响范围、继续补翻成本、精确重置成本和完整重译成本。未经用户明确选择，禁止执行 `reset-translations --all`，禁止删除全部已保存译文记录；优先只处理还没成功保存译文的文本，或用 `reset-translations --input <规则文件>` 精确重置受影响文本。
- 自定义占位符规则默认读取当前游戏数据库；CLI 显式传入自定义占位符规则 JSON 字符串时，本次运行只解析该字符串，不读取数据库规则。
- 自定义占位符规则不得塞进 `setting.toml`，也不得把项目根目录草稿文件当成主流程隐式数据源。
- 占位符规则较长时优先通过 `validate-placeholder-rules --input <规则文件>` 与 `import-placeholder-rules --input <规则文件>` 处理，避免把长 JSON 塞进命令行。
- 文本处理逻辑中的残留检测、长文本切行和提取阶段文本正规化，必须优先放进 `setting.toml` 的 `[text_rules]`。
- 禁止在业务代码里新增游戏风格相关的硬编码判断。
- RPG Maker MZ 引擎结构常量可以保留在代码中，例如事件命令编号 `101/401/102/405/357` 和标准 data 文件名。
- 占位符规则 JSON 必须使用对象格式，键为正则表达式字符串，值为占位符模板字符串。
- 自定义占位符模板必须生成形如 `[CUSTOM_NAME_1]` 的方括号占位符，推荐使用 `{index}` 保证同一条规则的多次命中可区分。
- 修改文本规则或自定义占位符规则时必须补充测试，至少覆盖默认 RMMZ 标准控制符和一组自定义正则占位符策略。

## 6. 日志与失败规范

- 业务层日志必须通过统一日志门面输出。
- 终端日志面向用户扫读，文件日志面向排障追踪。
- 未知异常的完整 traceback 必须写入文件日志，终端只输出中文摘要。
- 已知业务失败必须输出可理解的中文原因。
- 日志文案必须描述当前动作和当前结果，不得夹带变更说明。

## 7. 外部代理调用规范

- 外部代理执行翻译任务时，只能依赖本项目 Skill/规范、CLI 命令、`--json` 输出、工作区文件、当前游戏目录和用户明确提供的信息；不得依赖项目源码、数据库表结构、内部 Python 对象或实现细节来完成业务判断。
- 如果任务说明、CLI 输出或工作区说明信息不足，必须优先补充 Skill、CLI 输出、工作区说明或校验报告，而不是要求外部代理通过阅读源码猜测格式。
- 外部代理与子代理任务必须采用“输入 -> 逻辑 -> 输出”的任务契约表达。主代理在派发任务前必须明确：
  - **输入**：需要读取的文件、目录、CLI 输出、工作区文件和只读范围。
  - **逻辑**：筛选、判断、转换、排除、空结果判定和失败恢复规则。
  - **输出**：唯一可写文件、目标 JSON/文本格式、完成报告和后续校验命令。
- 禁止只用一句“分析这个文件”或“处理这些规则”派发任务；任务说明必须让接收方在不阅读项目源码的情况下知道拿什么输入、按什么规则工作、交付什么结果。
- 需要并行子代理时，主代理必须给每个子代理提供独立的任务契约，明确唯一写入边界，避免多个子代理写同一文件或基于隐含上下文自由发挥。
- 外部代理或子代理生成的文件必须先通过本项目 CLI 校验。校验失败时，优先根据错误修复对应外部 JSON 或工作区文件，再重新校验；禁止绕过 CLI 直接修改数据库或跳过校验进入翻译/写回。
- 需要 Claude Code 等外部代理处理项目文件时，默认在目标目录启动交互式会话。
- Claude Code 默认调用方式为 `claude --permission-mode bypassPermissions`，并在交互界面提交任务说明。
- 外部代理的工作目录必须是本次任务的最小目录，例如处理术语候选时进入对应游戏工作区的 `terminology` 目录。
- 交互式会话中必须让外部代理自行读取、搜索、分批分析和修改文件，不得把大量文件内容一次性塞进命令行 prompt。
- 只有用户明确要求非交互一次性执行时，才允许使用 `claude -p` 或同类 print 模式。
- 外部代理修改完成后，必须由本项目流程执行结构校验、关键字段校验和必要的差异检查。
- 外部代理在导出文件和导入文件之间可以使用自身工具、临时脚本或子代理处理数据；本项目只强制要求导出和导入必须使用本项目 CLI，并由本项目校验命令验收。
- 如果使用子代理并行处理术语候选、插件规则、事件指令规则或 Note 标签规则，主代理必须按两轮执行：第一轮先收集术语候选并亲自严审、修改、合并和导入术语表；第二轮再处理插件规则、事件指令规则和 Note 标签规则。占位符规则必须由主代理在两轮子代理任务完成后亲自处理。
- 处理 `terminology` 目录时，优先使用 `skills/att-mz/SKILL.md` 中的术语候选任务契约。
- 处理 `plugins.json` 时，优先使用 `docs/plugin-rules-agent-prompt.md` 中的参考提示词。
- 处理 `event-commands.json` 时，优先使用 `docs/event-command-rules-agent-prompt.md` 中的参考提示词。

## 8. Skill 编写规范

- 编写或修改 `SKILL.md` 时，必须写成可直接执行的任务契约：假设使用该 Skill 的代理不能阅读项目源码，不能理解数据库表结构，不能依赖内部实现细节，只能依赖 Skill 文档、CLI、`--json` 输出、工作区文件、用户输入和任务文件完成工作。
- Skill 不是项目说明书，也不是变更记录。Skill 必须是任务执行契约，写清代理在当前任务中拿什么输入、按什么逻辑处理、交付什么输出。
- Skill 中每个核心阶段、外部文件格式、子代理任务和关键命令，都必须能回答：
  - **输入**：读取哪些文件、命令输出、用户信息、工作区文件和只读范围。
  - **逻辑**：如何筛选、判断、转换、排除、处理空结果，以及失败时如何恢复。
  - **输出**：唯一可写位置、目标格式、完成报告、校验命令和进入下一阶段的条件。
- Skill 涉及外部代理或子代理时，必须提供可直接派发的任务契约。任务契约必须包含输入、逻辑、输出、唯一写入边界、空结果条件、完成报告和后续校验命令。
- Skill 涉及 JSON、规则文件、临时工作区或命令协议时，必须写清最小可用 schema、合法空结构、禁止字段、常见错误和校验失败恢复路径。禁止让代理通过阅读源码或试错来推断格式。
- Skill 中的命令流程必须写清前置条件、成功判断、什么情况不能继续，以及失败后的下一步。`--json` 输出的 `status=error` 表示当前阶段不能继续，不能进入后续阶段。
- 如果现有 Skill 无法让代理在不能阅读源码的条件下独立执行任务，应优先完善 Skill、CLI 输出或工作区说明，而不是要求代理读取源码补全理解。

## 8.1 发行版 Skill 与发布规范

- 源码开发任务使用 `skills/att-mz/SKILL.md`，发行版翻译任务使用 `skills/att-mz-release/SKILL.md` 作为模板；两个 Skill 的 frontmatter `name` 必须分别匹配各自文件夹名。
- 每次修改开发版 Skill 时，必须同步审查并更新发行版 Skill；发行版 Skill 必须使用 `.\att-mz.exe --agent-mode ...` 命令，不得要求用户安装 Python、Rust、uv 或读取项目源码。
- 每次修改发行版 Skill 时，也必须审查开发版 Skill 是否需要同步更新，避免两套任务协议语义漂移。
- 打包脚本必须把 `skills/att-mz-release/SKILL.md` 转换为发行包内的 `skills/att-mz/SKILL.md`，并把发行包内 frontmatter `name` 写成 `att-mz`；不得把开发版 Skill 原样放入发行包。
- 发行包不得包含源码数据库、历史日志、测试目录、Python 源码目录或 Rust 源码目录；只包含可执行文件、配置模板、README、许可证、字体、提示词、发行版 Skill、必要参考资料和空的数据/日志/输出目录。
- 每次正式发布发行版必须使用 GitHub Actions `release` 工作流生成并发布 ZIP，禁止在本机手工打包后直接上传 GitHub Release。
- 发行版构建脚本必须默认只允许在 GitHub Actions 环境中执行；本机只能提供源码改动、提交和工作流触发，不能作为发行版构建、压缩、冒烟测试或验收环境。
- 发行版验收必须由 GitHub Actions 发布工作流执行 `uv run basedpyright`、`uv run pytest`、发行包构建和发行包冒烟测试。

## 9. 交付检查

- 面向用户的文档、说明、示例命令、提示词模板和交付报告必须使用抽象占位符表达路径、游戏标题、临时目录和样例数据。
- 禁止在公开文档和通用说明中出现真实本机路径、真实用户名、真实样本游戏名、真实测试存档路径、真实测试数据定位路径或从用户机器提取出来的具体字段值。
- 示例命令必须使用 `<游戏标题>`、`<游戏根目录>`、`<外部临时目录>`、`<规则文件>` 这类占位符；确需展示格式时，只能使用构造的中性样例。
- 测试夹具中的文件名和定位路径只能服务于自动化测试，不得复制到 README、指南、提示词模板或对外说明中当作示例。
- 修改 prompt 组装逻辑后，必须用测试覆盖最终 user prompt 文本。
- prompt 测试必须断言不会出现内部文件名、内部字段名和 `位置:` 这类无效上下文。
- 修改文本规则、导入格式、数据库 schema、外部事件指令规则或回写逻辑时，必须补充对应测试。
