# 项目局部规范

本规范约束 `rpg-maker-tools` 仓库内的提示词、临时文件、插件规则、文本规则、日志和交付行为。未覆盖事项继续遵循用户级通用规范。

## 1. 提示词规范

- 给 LLM 的提示词只能包含模型完成当前任务所需的信息。
- 禁止把仅供程序定位、调试、读写文件的内部实现细节塞进 prompt。
- 禁止在 prompt 中暴露模型无法访问也无法使用的来源说明，例如 `name_registry.json`、数据库表名、内部字段名、文件路径、`location_path`、`translated_text`。
- 术语表必须使用翻译领域常用称呼 `[[术语表]]`。
- 术语表条目只允许呈现为 `原文 => 标准译名`。
- 术语表条目禁止附加位置、文件名、JSON 字段名、数据库主键等内部定位信息。
- 内部路径和定位信息只能用于程序筛选术语、写回 `101.parameters[4]`、写回 `MapXXX.displayName`、日志和排障。
- 只有当模型必须原样回传某个标识以便程序绑定输出时，才允许在 prompt 中放置内部 ID；内部 ID 必须放在明确的结构化输出协议中。

## 2. 外部临时文件规范

- `101` 名字框与 `MapXXX.displayName` 的译名由外部 Agent 填写大 JSON。
- 面向外部 Agent 的 JSON 必须使用最小结构：能用 `原文: 译名` 表达的内容不得拆成 ID、位置数组、类型字段和说明字段。
- 外部规则文件不得包含版本号、游戏标题、规则 ID、原因说明、内部路径、上下文文件名等主流程不需要读取的字段。
- `name_registry.json` 是项目外临时文件，必须通过 `import-name-context --game ... --input ...` 显式导入当前游戏数据库。
- 插件配置 JSON 是项目外临时文件，必须通过 `export-plugins-json --game ... --output ...` 从当前游戏导出。
- 插件配置 JSON 的顶层必须是 RPG Maker `$plugins` 数组本体，不得添加候选字段、说明字段或判断结果。
- 插件规则 JSON 是项目外临时文件，必须通过 `import-plugin-rules --game ... --input ...` 显式导入当前游戏数据库。
- 事件指令参数 JSON 是项目外临时文件，必须通过 `export-event-commands-json --game ... --output ...` 从当前游戏导出。
- 事件指令规则 JSON 是项目外临时文件，必须通过 `import-event-command-rules --game ... --input ...` 显式导入当前游戏数据库。
- 主流程的数据源必须是当前游戏数据库，`translate`、`write-back`、`run-all` 和提示词组装流程不得直接读取项目外临时 JSON。
- 多游戏数据必须按 `data/db/<游戏标题>.db` 分库存储。
- 不同游戏的术语表、插件规则和译文缓存不得混放在项目固定目录中作为运行时数据源。

## 3. 插件规则规范

- `plugins.js` 可翻译字段由外部 Agent 产出规则 JSON。
- 插件规则导入时必须校验插件名称、插件哈希和 JSONPath 字符串叶子命中结果。
- 插件规则导入后只能把校验后的规则条目写入数据库。
- 导入流程不得把外部临时文件路径保存为长期业务数据源。

## 4. 事件指令规则规范

- `data/*.json` 中的事件指令参数文本统一由外部 Agent 产出规则 JSON。
- 事件指令参数 JSON 未传 `--code` 时必须读取 `setting.toml` 的 `event_command_text.default_command_codes` 数组。
- 事件指令参数 JSON 传入 `--code` 时必须只使用 CLI 编码数组，并覆盖配置文件默认编码数组。
- 事件指令规则导入时必须校验指令编码、参数过滤条件和 JSONPath 字符串叶子命中结果。
- 事件指令规则必须显式声明 `command_code`，不得在业务代码中按固定编码猜测可翻译字段。
- 事件指令规则导入后只能把校验后的规则条目写入数据库。
- 主翻译流程只能按数据库事件指令规则提取复杂参数文本。

## 5. 文本规则配置规范

- RMMZ 标准文本控制符属于引擎协议，必须由代码内置并完整测试覆盖。
- 每次对真实游戏执行翻译测试、回写测试或质量验证前，必须先扫描当前游戏文本，找出完整的游戏自定义控制符、脚本标记和特殊占位符，并把对应自定义正则规则显式写入 `custom_placeholder_rules.json` 或通过 CLI `--placeholder-rules` 传入。
- 自定义占位符规则确认完成前，禁止启动会消耗模型额度的正文翻译任务。
- 自定义占位符规则默认读取项目根目录的 `custom_placeholder_rules.json`，也可以通过 `translate --placeholder-rules ...` 或 `run-all --placeholder-rules ...` 直接传入本次运行使用的 JSON 字符串。
- CLI 显式传入自定义占位符规则 JSON 字符串时，本次运行只解析该字符串。
- 自定义占位符规则不得塞进 `setting.toml`。
- 文本处理逻辑中的残留检测、长文本切行和提取阶段文本正规化，必须优先放进 `setting.toml` 的 `[text_rules]`。
- 禁止在业务代码里新增游戏风格相关的硬编码判断。
- RPG Maker MZ 引擎结构常量可以保留在代码中，例如事件命令编号 `101/401/102/405/357` 和标准 data 文件名。
- `custom_placeholder_rules.json` 必须使用 JSON 对象格式，键为正则表达式字符串，值为占位符模板字符串。
- 自定义占位符模板必须生成形如 `[CUSTOM_NAME_1]` 的方括号占位符，推荐使用 `{index}` 保证同一条规则的多次命中可区分。
- 修改文本规则或自定义占位符规则时必须补充测试，至少覆盖默认 RMMZ 标准控制符和一组自定义正则占位符策略。

## 6. 日志与失败规范

- 业务层日志必须通过统一日志门面输出。
- 终端日志面向用户扫读，文件日志面向排障追踪。
- 未知异常的完整 traceback 必须写入文件日志，终端只输出中文摘要。
- 已知业务失败必须输出可理解的中文原因。
- 日志文案必须描述当前动作和当前结果，不得夹带变更说明。

## 7. 外部 Agent 调用规范

- 需要 Claude Code 等外部 Agent 处理项目文件时，默认在目标目录启动交互式会话。
- Claude Code 默认调用方式为 `claude --permission-mode bypassPermissions`，并在交互界面提交任务说明。
- 外部 Agent 的工作目录必须是本次任务的最小目录，例如处理术语表时进入对应游戏的 `name-context` 目录。
- 交互式会话中必须让外部 Agent 自行读取、搜索、分批分析和修改文件，不得把大量文件内容一次性塞进命令行 prompt。
- 只有用户明确要求非交互一次性执行时，才允许使用 `claude -p` 或同类 print 模式。
- 外部 Agent 修改完成后，必须由本项目流程执行结构校验、关键字段校验和必要的差异检查。
- 处理 `name-context` 目录时，优先使用 `docs/name-context-agent-prompt.md` 中的参考提示词。
- 处理 `plugins.json` 时，优先使用 `docs/plugin-rules-agent-prompt.md` 中的参考提示词。
- 处理 `event-commands.json` 时，优先使用 `docs/event-command-rules-agent-prompt.md` 中的参考提示词。

## 8. 交付检查

- 面向用户的文档、说明、示例命令、提示词模板和交付报告必须使用抽象占位符表达路径、游戏标题、临时目录和样例数据。
- 禁止在公开文档和通用说明中出现真实本机路径、真实用户名、真实样本游戏名、真实测试存档路径、真实测试数据定位路径或从用户机器提取出来的具体字段值。
- 示例命令必须使用 `<游戏标题>`、`<游戏根目录>`、`<外部临时目录>`、`<规则文件>` 这类占位符；确需展示格式时，只能使用人工构造的中性样例。
- 测试夹具中的文件名和定位路径只能服务于自动化测试，不得复制到 README、指南、提示词模板或对外说明中当作示例。
- 修改 prompt 组装逻辑后，必须用测试覆盖最终 user prompt 文本。
- prompt 测试必须断言不会出现内部文件名、内部字段名和 `位置:` 这类无效上下文。
- 修改文本规则、导入格式、数据库 schema、外部事件指令规则或回写逻辑时，必须补充对应测试。
- 交付前必须执行 `uv run basedpyright` 和 `uv run pytest`，保持 0 warning、0 error。
